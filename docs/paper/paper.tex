\begin{abstract}
We propose NARAD, an architecture for end-to-end verifiable electronic voting that combines Paillier additive homomorphic encryption with blockchain immutability. The system follows an aggregator-oblivious paradigm where voters encrypt their ballots independently using self-generated secret keys---without any key coordination---and an aggregator computes the tally without learning individual votes. We present the cryptographic protocol with formal correctness proofs under six explicitly stated assumptions, analyze the security properties under the Decisional Composite Residuosity assumption, and detail a containerized microservices design where encryption runs client-side in WebAssembly and aggregation executes via a native C library (libtommath). A proof-of-concept implementation on Solana demonstrates exact vote recovery for up to 33.5 million voters across 10 candidates with sub-second aggregation latency.
\end{abstract}

\medskip
\noindent\textbf{Keywords:} Paillier cryptosystem, homomorphic encryption, blockchain voting, aggregator-oblivious, WebAssembly, libtommath, Solana, microservices

\section{Introduction}
% ============================================================

Electronic voting must reconcile two seemingly irreconcilable properties: \emph{ballot privacy} (no one should learn how an individual voted) and \emph{tally verifiability} (anyone should be able to verify the announced result is correct). Classical approaches trade one for the other. Homomorphic encryption offers a principled resolution: the tally is computed on ciphertexts without decrypting individual ballots.

\subsection{Problem Statement}

We seek a voting system satisfying the following:

\begin{itemize}[leftmargin=*]
    \item \textbf{Privacy:} No party---aggregator, collector, or network observer---learns any individual vote, even if all parties collude except the voter.
    \item \textbf{Verifiability:} The final tally is publicly verifiable. Each voter can confirm their encrypted ballot was recorded.
    \item \textbf{No key coordination:} Voters generate their own secret keys without a dealer. This eliminates a trust bottleneck.
    \item \textbf{Practicality:} The system should handle millions of voters and multiple candidates, with sub-minute aggregation.
\end{itemize}

\subsection{Approach}

We adopt the aggregator-oblivious encryption scheme~\cite{lem14b}: a trusted third party sets up the cryptographic parameters but does not know individual secret keys. Voters encrypt independently and produce auxiliary values that allow the aggregator to cancel the random mask in the homomorphic product. The key insight is that the voter's secret key $sk_i$ serves double duty---as the encryption randomness and as the exponent in the auxiliary value---so the mask $H^{\sum sk_i}$ is cancelled by $aux = H^{sk_A \cdot \sum sk_i}$ during aggregation.

\subsection{Contributions}

\begin{enumerate}[leftmargin=*]
    \item We \textbf{propose} a microservices architecture for blockchain voting that separates cryptographic computation (WASM + native C) from election management (Solana blockchain) and data persistence (PostgreSQL).
    \item We provide \textbf{formal correctness proofs} for the full aggregation pipeline under six explicitly stated assumptions, including the binomial expansion property, mask cancellation, and slot independence.
    \item We analyze \textbf{security} under the Decisional Composite Residuosity (DCR) assumption and identify five limitations of the current design.
    \item We describe a \textbf{proof-of-concept implementation} and stress test results demonstrating exact vote recovery at scale.
\end{enumerate}

\section{Proposed Architecture}\label{sec:architecture}
% ============================================================

The system is organized as a set of loosely-coupled services, each with a single responsibility. Figure~\ref{fig:architecture} shows the overall design.

\begin{figure}[h]
\centering
\includegraphics[width=\textwidth]{figures/system_architecture.pdf}
\caption{Proposed microservices architecture. Client-side encryption runs in the browser via WebAssembly. The backend coordinates between PostgreSQL (parameter and vote storage) and Solana (on-chain immutability). The native libtommath module performs aggregation. Dashed lines indicate trust boundaries.}
\label{fig:architecture}
\end{figure}

\subsection{Components}

\subsubsection{Voter Client (Browser + WASM)}
The voter interacts with a web portal. A WebAssembly module---compiled from C using Emscripten with libtommath---performs all cryptographic operations \emph{client-side}: key generation, vote packing, Paillier encryption, and auxiliary value computation. The plaintext vote and the voter's secret key $sk_i$ never leave the browser. The WASM module exports functions callable from JavaScript:

\begin{itemize}[leftmargin=*]
    \item \texttt{generate\_secret\_key(N)} $\to$ $sk_i$
    \item \texttt{pack\_votes}($\mathbf{v}$, $k$) $\to$ $m$
    \item \texttt{encrypt\_vote\_paillier}($m$, $H$, $N$) $\to$ $c_i$
    \item \texttt{compute\_auxiliary\_key}($pk_A$, $sk_i$, $N$) $\to$ $aux_i$
\end{itemize}

\subsubsection{Backend API (Node.js)}
An Express server exposing REST endpoints for voter authentication, election management, and aggregation. It enforces role-based access control (RBAC) via JWT tokens: \texttt{admin} role for election management, \texttt{voter} role for casting ballots. The backend is the \emph{orchestrator}---it does not perform cryptographic computation itself but delegates to the native module and the blockchain.

\subsubsection{Aggregation Engine (Native C / libtommath)}
A Node.js native addon compiled from C, linking against libtommath for arbitrary-precision integer arithmetic. This is the performance-critical component: all modular exponentiations and inversions are performed in C for speed. The addon exposes \texttt{aggregatorInit}, \texttt{addCiphertextToProduct}, and \texttt{aggregateVotesFromRunningProduct} to the backend.

\subsubsection{Database (PostgreSQL)}
Stores three tables: \texttt{system\_params} ($N$, $H$, $sk_A$), \texttt{voter\_data} (per-voter $c_i$ and $aux_i$), and \texttt{aggregated\_results} (the final tally). The database is the \textbf{single source of truth} for cryptographic parameters---not the blockchain. This is a deliberate design decision: parameters may need updating (e.g., replacing an invalid $N$), and the blockchain is immutable.

\subsubsection{Blockchain (Solana)}
Provides on-chain election lifecycle management: election creation, candidate whitelisting, stage transitions (application $\to$ voting $\to$ closed), and immutable vote recording. Each encrypted ballot $c_i$ is submitted as a Solana transaction, creating an append-only audit trail. The blockchain does \emph{not} participate in aggregation---it provides immutability, not computation.

\subsection{Design Rationale}

\paragraph{Why WASM for encryption?} The Paillier encryption requires modular exponentiation on 510-bit numbers. JavaScript's \texttt{BigInt} is correct but slow for repeated exponentiation. C compiled to WASM via Emscripten, using libtommath's optimized \texttt{mp\_exptmod}, provides native-speed arithmetic in the browser sandbox.

\paragraph{Why a native addon for aggregation?} The aggregation involves $n$ modular multiplications and one large exponentiation. For $n = 30$ million, the difference between JS BigInt and native C is 3--5$\times$. The native addon links against the same libtommath library used in the WASM module, ensuring consistency.

\paragraph{Why separate crypto params from blockchain?} The Solana program stores $N$, $H$, $sk_A$ at election creation time. If a parameter is found to be invalid (e.g., $N$ is even), updating the blockchain requires redeploying the program. By storing parameters in PostgreSQL, we can update them without blockchain interaction. The blockchain stores election metadata (candidates, stages) for immutability, while the database stores cryptographic parameters for flexibility.

\paragraph{Why microservices?} Each component can be scaled, deployed, and updated independently. The backend can run multiple replicas behind a load balancer. The database can use streaming replication. The Solana validator runs independently. Containerization (Docker) ensures reproducible deployments.

\section{Cryptographic Protocol}\label{sec:protocol}
% ============================================================

The protocol involves four parties: a trusted third party $TP$, voters $U_1, \ldots, U_n$, a collector $C$, and an aggregator $A$. Figure~\ref{fig:protocol} shows the full execution flow.

\begin{figure}[h]
\centering
\includegraphics[width=\textwidth]{figures/protocol_flow.pdf}

\subsection{Assumptions}

\begin{assumption}[Valid Modulus]\label{ass:modulus}
$N = p \cdot q$ where $p$ and $q$ are distinct odd primes. This ensures $\mathbb{Z}_{N^2}^*$ is well-defined, $\gcd(H, N^2) = 1$ for random $H \in \mathbb{Z}_{N^2}^*$, and modular inverses exist.
\end{assumption}

\begin{assumption}[Non-Degenerate Key]\label{ass:key}
$sk_A \neq H$ and $\gcd(sk_A, N) = 1$. This ensures $pk_A = H^{sk_A} \neq H$ (non-trivial public key) and $sk_A^{-1} \bmod N$ exists (for sum recovery).
\end{assumption}

\begin{assumption}[Sum Bound]\label{ass:sum}
$S = \sum_{i=1}^{n} x_i < N$. This ensures the modular reduction in the final step is exact: $R = S \bmod N = S$ (no wraparound).
\end{assumption}

\begin{assumption}[Packing Capacity]\label{ass:packing}
$k \cdot b < \log_2 N$ where $k$ is the number of candidates and $b = 25$ is the bits per slot. This ensures the packed vote $m$ fits in a single Paillier plaintext space.
\end{assumption}

\begin{assumption}[Binary Votes]\label{ass:binary}
Each $v_{i,j} \in \{0, 1\}$ and the total per candidate $\sum_i v_{i,j} < 2^b$. This ensures no slot overflow during homomorphic addition.
\end{assumption}

\begin{assumption}[Key as Randomness]\label{ass:keyrand}
The voter's secret key $sk_i$ is used as the encryption randomness: $r_i = sk_i$. This is the key insight that makes the mask cancellation work: $aux_i = pk_A^{sk_i} = H^{sk_A \cdot sk_i}$ cancels the $H^{sk_i}$ term in the ciphertext product when raised to $sk_A$.
\end{assumption}

\subsection{Setup Phase}

$TP$ selects safe primes $p, q$, computes $N = pq$, and picks $H \in_R \mathbb{Z}_{N^2}^*$. $TP$ publishes $(N, H)$ and goes offline.

Aggregator $A$ generates $sk_A \in_R \mathbb{Z}_{N^2}^*$ and publishes $pk_A = H^{sk_A} \bmod N^2$.

Each voter $U_i$ independently generates $sk_i \in_R [0, N)$.

\subsection{Encryption Phase}

Voter $U_i$ encodes their ballot as a packed integer $x_i$ (Section~\ref{sec:packing}) and computes:
\begin{align}
c_i &= H^{sk_i} \cdot (1 + N)^{x_i} \bmod N^2 \label{eq:enc}\\
aux_i &= pk_A^{sk_i} = H^{sk_A \cdot sk_i} \bmod N^2 \label{eq:auxenc}
\end{align}

$c_i$ is sent to the aggregator; $aux_i$ is sent to the collector.

\begin{remark}
Eq.~\ref{eq:enc} uses $(1+N)^{x_i}$ rather than the equivalent $(1 + x_i \cdot N)$ for $x_i < N$. Both are equal in $\mathbb{Z}_{N^2}$ by the binomial theorem: $(1+N)^{x_i} \equiv 1 + x_i \cdot N \pmod{N^2}$. The former is used in the implementation for consistency with standard Paillier notation.
\end{remark}

\subsection{Collection Phase}

Collector $C$ computes:
\begin{equation}
aux = \prod_{i=1}^{n} aux_i = H^{sk_A \cdot \sum_{i=1}^n sk_i} \bmod N^2 \label{eq:collect2}
\end{equation}

$aux$ is sent to the aggregator.

\subsection{Aggregation Phase}

The aggregator computes the sum in five steps (Section~\ref{sec:proofs} proves correctness):

\begin{align}
\Pi &= \prod_{i=1}^{n} c_i \bmod N^2 \label{eq:step1}\\
P &= \Pi^{sk_A} \bmod N^2 \label{eq:step2}\\
P' &= P \cdot aux^{-1} \bmod N^2 \label{eq:step3}\\
L(P') &= \frac{P' - 1}{N} \bmod N \label{eq:step4}\\
R &= L(P') \cdot sk_A^{-1} \bmod N \label{eq:step5}
\end{align}

Under Assumptions~\ref{ass:modulus}--\ref{ass:sum}, $R = \sum_{i=1}^n x_i$ exactly.

\section{Correctness Proofs}\label{sec:proofs}
% ============================================================

\begin{lemma}[Binomial Expansion in $\mathbb{Z}_{N^2}$]\label{lem:binom}
For $|a_i| < N$ and $i = 1, \ldots, n$:
$$\prod_{i=1}^{n} (1 + a_i N) \equiv 1 + \left(\sum_{i=1}^{n} a_i\right) N \pmod{N^2}$$
\end{lemma}

\begin{proof}
By induction on $n$. Base case ($n=1$) is trivial. For $n = 2$:
$(1 + a_1 N)(1 + a_2 N) = 1 + (a_1 + a_2)N + a_1 a_2 N^2 \equiv 1 + (a_1 + a_2)N \pmod{N^2}$

Inductive step: assume for $n-1$. Then:
$\prod_{i=1}^{n}(1+a_iN) = \left(1 + \left(\sum_{i=1}^{n-1} a_i\right)N\right)(1 + a_n N) = 1 + \left(\sum_{i=1}^{n} a_i\right)N + \left(\sum_{i=1}^{n-1}a_i\right)a_n N^2 \equiv 1 + \left(\sum_{i=1}^n a_i\right)N \pmod{N^2}$

The $N^2$ term vanishes modulo $N^2$.
\end{proof}

\begin{theorem}[Ciphertext Product Homomorphism]\label{thm:product}
Under Assumptions~\ref{ass:modulus} and \ref{ass:packing}, the product of ciphertexts satisfies:
$$\prod_{i=1}^{n} c_i = \left(1 + S \cdot N\right) \cdot H^{\Sigma} \bmod N^2$$
where $S = \sum_{i=1}^n x_i$ and $\Sigma = \sum_{i=1}^n sk_i$.
\end{theorem}

\begin{proof}
From Eq.~\ref{eq:enc}: $c_i = H^{sk_i} \cdot (1+N)^{x_i} \bmod N^2$. By the binomial theorem, $(1+N)^{x_i} \equiv 1 + x_i N \pmod{N^2}$ for $x_i < N$ (Assumption~\ref{ass:packing}). So $c_i = H^{sk_i} \cdot (1 + x_i N) \bmod N^2$.

Taking the product:
$$\prod c_i = \left(\prod H^{sk_i}\right) \cdot \left(\prod (1 + x_i N)\right) = H^{\Sigma} \cdot \prod (1 + x_i N) \bmod N^2$$

By Lemma~\ref{lem:binom}, $\prod(1 + x_i N) \equiv 1 + S \cdot N \pmod{N^2}$. Therefore:
$$\prod c_i = (1 + S \cdot N) \cdot H^{\Sigma} \bmod N^2 \qedhere$$
\end{proof}

\begin{theorem}[Mask Cancellation]\label{thm:mask}
Under Assumptions~\ref{ass:modulus} and \ref{ass:keyrand}, the mask is exactly cancelled:
$$P' = P \cdot aux^{-1} = (1 + S \cdot N)^{sk_A} \bmod N^2$$
\end{theorem}

\begin{proof}
From Theorem~\ref{thm:product}, $\Pi = (1 + SN) \cdot H^{\Sigma} \bmod N^2$. Then:
$$P = \Pi^{sk_A} = (1+SN)^{sk_A} \cdot H^{sk_A \cdot \Sigma} \bmod N^2$$

From Eq.~\ref{eq:collect2}, $aux = H^{sk_A \cdot \Sigma} \bmod N^2$. By Assumption~\ref{ass:modulus}, $N$ is a valid semiprime, so $\gcd(H, N^2) = 1$, and $aux^{-1} \bmod N^2$ exists. Therefore:
$$P' = P \cdot aux^{-1} = (1+SN)^{sk_A} \cdot H^{sk_A \Sigma} \cdot H^{-sk_A \Sigma} = (1+SN)^{sk_A} \bmod N^2 \qedhere$$
\end{proof}

\begin{theorem}[Sum Recovery]\label{thm:sum}
Under Assumptions~\ref{ass:modulus}, \ref{ass:key}, and \ref{ass:sum}:
$$R = L(P') \cdot sk_A^{-1} \bmod N = S = \sum_{i=1}^n x_i$$
\end{theorem}

\begin{proof}
By the binomial theorem, $(1+SN)^{sk_A} \equiv 1 + sk_A \cdot S \cdot N \pmod{N^2}$ (higher-order terms contain $N^2$ and vanish). Thus:
$$L(P') = \frac{(1 + sk_A \cdot S \cdot N) - 1}{N} = sk_A \cdot S$$

By Assumption~\ref{ass:key}, $\gcd(sk_A, N) = 1$, so $sk_A^{-1} \bmod N$ exists. Therefore:
$$R = sk_A \cdot S \cdot sk_A^{-1} \bmod N = S \bmod N$$

By Assumption~\ref{ass:sum}, $S < N$, so $S \bmod N = S$.
\end{proof}

\begin{corollary}[Vote Count Correctness]\label{cor:votes}
Under Assumptions~\ref{ass:packing} and \ref{ass:binary}, the extracted vote count for candidate $j$ equals the true tally:
$$\text{count}_j = \sum_{i=1}^{n} v_{i,j}$$
\end{corollary}

\begin{proof}
By Theorem~\ref{thm:sum}, $R = \sum x_i$. Each $x_i = \sum_j v_{i,j} \cdot 2^{b(k-j)}$, so:
$$R = \sum_i \sum_j v_{i,j} \cdot 2^{b(k-j)} = \sum_j \underbrace{\left(\sum_i v_{i,j}\right)}_{T_j} \cdot 2^{b(k-j)}$$

The slot extraction $\lfloor R / 2^{b(k-j)} \rfloor \bmod 2^b$ isolates $T_j$, provided $T_j < 2^b$ (Assumption~\ref{ass:binary}). Since each $v_{i,j} \in \{0,1\}$, we have $T_j = \sum_i v_{i,j}$, which is the true vote count for candidate $j$.
\end{proof}

\section{Security Analysis}\label{sec:security}
% ============================================================

\subsection{Cryptographic Assumption}

The security of the Paillier cryptosystem rests on the \textbf{Decisional Composite Residuosity (DCR) assumption}~\cite{paillier1999public}:

\begin{definition}[DCR]
Given $N = pq$ and $z \in \mathbb{Z}_{N^2}^*$, it is computationally infeasible to distinguish whether $z$ is an $N$-th residue modulo $N^2$ (i.e., $\exists y: z \equiv y^N \pmod{N^2}$) or a random element of $\mathbb{Z}_{N^2}^*$.
\end{definition}

Under DCR, the encryption $c_i = H^{sk_i} \cdot (1+N)^{x_i} \bmod N^2$ is \emph{semantically secure}: an adversary cannot distinguish $c_i$ from a random element of $\mathbb{Z}_{N^2}^*$ without knowing the factorization of $N$.

\subsection{Privacy Analysis}

\begin{proposition}[Individual Vote Privacy]\label{prop:privacy}
Under DCR, no computationally bounded adversary seeing $c_i$ can learn $x_i$ with non-negligible advantage, even given $(N, H, pk_A)$.
\end{proposition}

\begin{proof}
$c_i = H^{sk_i} \cdot (1+N)^{x_i} \bmod N^2$. The term $H^{sk_i}$ is a random element of $\mathbb{Z}_{N^2}^*$ (since $H$ is a generator and $sk_i$ is random). The term $(1+N)^{x_i}$ is the Paillier encryption of $x_i$. By the semantic security of Paillier under DCR~\cite{paillier1999public}, $c_i$ is indistinguishable from random.
\end{proof}

\begin{proposition}[Aggregator Obliviousness]\label{prop:agg}
The aggregator $A$, seeing $\{c_i\}_{i=1}^n$ and $aux$, learns only $S = \sum x_i$ and nothing about individual $x_i$ values.
\end{proposition}

\begin{proof}
$A$ computes $R = S$ from the protocol. To learn $x_i$ individually, $A$ would need to decrypt $c_i$, which requires either the factorization of $N$ (hard under the integer factorization assumption) or solving DCR. The value $aux = H^{sk_A \cdot \sum sk_i}$ is a single group element that encodes collective information; it does not reveal individual $sk_i$ values under DCR.
\end{proof}

\begin{proposition}[Collector Obliviousness]\label{prop:collector}
The collector $C$, seeing $\{aux_i\}_{i=1}^n$, learns nothing about any $x_i$.
\end{proposition}

\begin{proof}
Each $aux_i = pk_A^{sk_i} = H^{sk_A \cdot sk_i} \bmod N^2$ is a random element of $\mathbb{Z}_{N^2}^*$ (since $H$ is a generator and $sk_A \cdot sk_i$ is effectively random). Under DCR, these are indistinguishable from random. The collector sees no ciphertexts $c_i$ and thus has no information about votes.
\end{proof}

\subsection{Integrity Properties}

\textbf{On-chain immutability.} Each $c_i$ is submitted as a Solana transaction. Once finalized, the on-chain record cannot be altered. This provides a verifiable audit trail: any discrepancy between the database and the blockchain can be detected.

\textbf{Stage enforcement.} The Solana program enforces election stages: votes are only accepted during the Voting stage, candidates only during Application. This prevents premature voting or late candidate addition.

\textbf{RBAC.} Sensitive operations require admin JWT. Voters cannot create elections or modify candidates.

\subsection{Limitations}

\begin{enumerate}[leftmargin=*]
    \item \textbf{No range proofs:} The system does not verify $v_{i,j} \in \{0,1\}$. A malicious voter could submit a large value. Zero-knowledge range proofs would address this.
    \item \textbf{Trusted setup:} $TP$ knows the factorization of $N$. Distributed key generation would remove this trust.
    \item \textbf{Single aggregator:} $A$ knows $sk_A$ and could fake results. Threshold decryption across multiple aggregators would mitigate this.
    \item \textbf{No coercion resistance:} A voter can prove how they voted by revealing $sk_i$. Receipt-freeness mechanisms~\cite{kiayias2008civitas} would be needed.
    \item \textbf{Local blockchain:} The test validator does not provide Byzantine fault tolerance. Mainnet deployment is needed for real security.
\end{enumerate}

\section{Vote Packing}\label{sec:packing}
% ============================================================

To aggregate a multi-candidate election in a single Paillier ciphertext, votes are packed into a single integer using fixed-width bit slots. For $k$ candidates with $b = 25$ bits per slot:

\begin{equation}
x_i = \sum_{j=1}^{k} v_{i,j} \cdot 2^{b(k-j)} = v_{i,1} \cdot 2^{b(k-1)} + v_{i,2} \cdot 2^{b(k-2)} + \cdots + v_{i,k} \cdot 2^{0}
\end{equation}

Candidate 1 occupies the \emph{most significant} slot (bits $b(k-1)$ to $bk-1$), and candidate $k$ occupies the \emph{least significant} slot (bits 0 to $b-1$). This reverse ordering is an implementation choice; the key requirement is consistency between packing and unpacking.

The homomorphic sum preserves slots independently because addition in $\mathbb{Z}$ is slot-wise when no slot overflows:
\begin{equation}
\sum_{i=1}^n x_i = \sum_{j=1}^{k} \left(\sum_{i=1}^n v_{i,j}\right) \cdot 2^{b(k-j)}
\end{equation}

Extraction: $\text{count}_j = \lfloor R / 2^{b(k-j)} \rfloor \bmod 2^b$.

Figure~\ref{fig:packing} illustrates the layout for $k = 10$, $b = 25$.

\begin{figure}[h]
\centering
\includegraphics[width=\textwidth]{figures/vote_packing.pdf}
\caption{25-bit vote packing for 10 candidates. Each slot occupies 25 bits. Total: $10 \times 25 = 250$ bits $< 255 = \lceil\log_2 N\rceil$. Maximum votes per candidate: $2^{25} - 1 = 33{,}554{,}431$.}
\label{fig:packing}
\end{figure}

\subsection{Capacity Analysis}

The system capacity is determined by Assumptions~\ref{ass:packing} and \ref{ass:binary}:

\begin{itemize}[leftmargin=*]
    \item \textbf{Max candidates:} $k_{\max} = \lfloor \log_2 N / b \rfloor = \lfloor 255 / 25 \rfloor = 10$
    \item \textbf{Max votes per candidate:} $2^b - 1 = 33{,}554{,}431$
    \item \textbf{Max total voters:} $k_{\max} \times (2^b - 1) \approx 335.5$ million
\end{itemize}

For more candidates, a larger $N$ is needed. Table~\ref{tab:scaling} shows scaling requirements.

\section{Algorithms}\label{sec:algorithms}
% ============================================================

\begin{algorithm}[h]
\caption{Voter Encryption (client-side, WASM)}
\label{alg:encrypt}
\begin{algorithmic}[1]
\Require Vote vector $\mathbf{v} = [v_1, \ldots, v_k] \in \{0,1\}^k$, public params $(N, H)$, $pk_A$
\Ensure Ciphertext $c$, auxiliary value $aux$

\Statex \textit{Assumptions:} \ref{ass:packing} ($k \cdot 25 < \log_2 N$), \ref{ass:binary} ($v_j \in \{0,1\}$), \ref{ass:keyrand} ($sk = r$)

\State $sk \xleftarrow{\$} [2, N-1]$ \Comment{Generate secret key = randomness}
\State $m \gets 0$
\For{$j = 1$ to $k$}
    \State $m \gets m + v_j \cdot 2^{25(k-j)}$ \Comment{Pack in reverse order}
\EndFor
\State $c \gets \textsc{ModExp}(H, sk, N^2) \cdot \textsc{ModExp}(1+N, m, N^2) \bmod N^2$
\State $aux \gets \textsc{ModExp}(pk_A, sk, N^2)$
\State \Return $(c, aux)$
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[h]
\caption{Vote Aggregation (server-side, native libtommath)}
\label{alg:aggregate}
\begin{algorithmic}[1]
\Require Ciphertexts $\{c_1, \ldots, c_n\}$, $aux$, params $(N, H, sk_A)$, candidate count $k$
\Ensure Vote counts $[\text{count}_1, \ldots, \text{count}_k]$

\Statex \textit{Assumptions:} \ref{ass:modulus} ($N=pq$, odd), \ref{ass:key} ($\gcd(sk_A,N)=1$), \ref{ass:sum} ($S < N$)

\State $N^2 \gets N \cdot N$
\State $sk_A^{-1} \gets \textsc{ModInv}(sk_A, N)$ \Comment{Exists by Assumption~\ref{ass:key}}

\Statex \Comment{\textbf{Step 1: Product of ciphertexts}}
\State $\Pi \gets 1$
\For{$i = 1$ to $n$}
    \State $\Pi \gets \Pi \cdot c_i \bmod N^2$ \Comment{\texttt{mp\_mul} + \texttt{mp\_mod}}
\EndFor

\Statex \Comment{\textbf{Step 2: Raise to $sk_A$}}
\State $P \gets \textsc{ModExp}(\Pi, sk_A, N^2)$ \Comment{\texttt{mp\_exptmod}}

\Statex \Comment{\textbf{Step 3: Cancel mask}}
\State $aux^{-1} \gets \textsc{ModInv}(aux, N^2)$ \Comment{Exists by Assumption~\ref{ass:modulus}}
\State $P' \gets P \cdot aux^{-1} \bmod N^2$

\Statex \Comment{\textbf{Step 4: L-function}}
\State $L \gets (P' - 1) \, \mathbf{div} \, N$ \Comment{Exact integer division}

\Statex \Comment{\textbf{Step 5: Recover sum}}
\State $R \gets L \cdot sk_A^{-1} \bmod N$ \Comment{Now $R = S = \sum x_i$}

\Statex \Comment{\textbf{Step 6: Unpack votes}}
\For{$j = 1$ to $k$}
    \State $\text{count}_j \gets (R \gg 25(k-j)) \mathbin{\&} (2^{25}-1)$ \Comment{Bit shift + mask}
\EndFor
\State \Return $[\text{count}_1, \ldots, \text{count}_k]$
\end{algorithmic}
\end{algorithm}

\subsection{Performance Characteristics}

\paragraph{Encryption throughput.} Approximately 2{,}000--3{,}000 votes/second with 16 parallel workers. Bottleneck: modular exponentiation on 510-bit numbers.

\paragraph{Aggregation latency.} Sub-second for 5{,}000 voters. Scales as $O(n)$ for the product step (one \texttt{mp\_mul} per ciphertext) and $O(1)$ for the decryption step (fixed-size exponentiation regardless of $n$).

\paragraph{Capacity.} 33.5 million voters (10 candidates $\times$ $2^{25}-1$ per slot), limited by Assumption~\ref{ass:packing}.

\section{Proof-of-Concept Implementation}\label{sec:implementation}
% ============================================================

The proposed architecture has been implemented as a proof-of-concept to validate the protocol. We briefly describe the key components; full source code is available at \url{https://github.com/Akshit11318/narad}.

\subsection{Technology Stack}

\begin{itemize}[leftmargin=*]
    \item \textbf{WASM:} C compiled with Emscripten 4.0.6, linking libtommath
    \item \textbf{Backend:} Node.js 20, Express, Prisma ORM
    \item \textbf{Native addon:} C + Node-API, cmake-js build, libtommath
    \item \textbf{Blockchain:} Solana (Anchor 0.30.1), test validator
    \item \textbf{Database:} PostgreSQL 16
    \item \textbf{Frontend:} React 19, Tailwind CSS
    \item \textbf{Deployment:} Docker Compose (4 containers)
\end{itemize}

\subsection{Engineering Challenges Resolved}

During implementation, seven critical bugs were identified and fixed:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Even modulus:} Hardcoded $N$ was even ($N \bmod 2 = 0$), breaking modular inverses. Fixed by generating proper semiprime $N = pq$.
    \item \textbf{Degenerate key:} $sk_A = H$ produced trivial decryption. Fixed with separate random $sk_A$.
    \item \textbf{Parameter source mismatch:} Aggregation read $N$ from blockchain (stale) instead of database (current). Fixed by using database as single source of truth.
    \item \textbf{Bit-packing mismatch:} C used 25-bit slots, JS fallback used 32-bit. Unified to 25-bit.
    \item \textbf{Missing $sk_A^{-1}$ step:} JS computed $L(P')$ but not $L(P') \cdot sk_A^{-1}$. Added the missing step.
    \item \textbf{C GCD pre-check bug:} Faulty GCD check rejected valid $sk_A$. Removed pre-check; \texttt{mp\_invmod} handles errors.
    \item \textbf{Byte-order mismatch:} C binding produced big-endian bytes; codebase expected little-endian. Added byte reversal.
\end{enumerate}

These bugs illustrate the gap between protocol design and implementation---particularly the subtlety of byte ordering in native bindings and the importance of a single source of truth for parameters.

\subsection{Docker Deployment}

All services run in Docker containers on a shared bridge network:
\begin{itemize}[leftmargin=*]
    \item \texttt{narad-postgres}: PostgreSQL 16 Alpine
    \item \texttt{narad-solana}: Solana test validator (tchambard/solana-test-validator)
    \item \texttt{narad-backend}: Node.js with native aggregator addon
    \item \texttt{narad-portal}: nginx serving React build
\end{itemize}

The backend auto-generates a Solana wallet on first startup and airdrops 100 SOL. The native aggregator is built inside the Docker image using cmake-js.

\section{Related Work}\label{sec:related}
% ============================================================

\paragraph{Helios~\cite{adida2008helios}} is a web-based open-audit voting system using homomorphic encryption. It relies on a trusted authority for decryption and does not use blockchain.

\paragraph{Civitas~\cite{kiayias2008civitas}} extends Helios with coercion resistance using distributed plaintext equivalence tests. NARAD does not address coercion resistance.

\paragraph{Aggregator-oblivious encryption~\cite{lem14b}} describes the protocol where users encrypt without key coordination. NARAD implements this protocol end-to-end.

\paragraph{Privacy-preserving aggregation~\cite{popets2023}} surveys aggregation protocols with formal privacy guarantees. NARAD's protocol is an instance of this family.

\section{Conclusion}\label{sec:conclusion}
% ============================================================

We proposed NARAD, a microservices architecture for privacy-preserving blockchain voting based on Paillier homomorphic encryption. The key insight is that voters use their self-generated secret key as both the encryption randomness and the auxiliary value exponent, enabling mask cancellation without key coordination. We proved correctness under six explicit assumptions and analyzed security under DCR. A proof-of-concept implementation confirmed exact vote recovery at scale.

Future work includes: (1) zero-knowledge range proofs for vote validity, (2) threshold decryption across multiple aggregators, (3) distributed key generation to eliminate trusted setup, and (4) Solana mainnet deployment with program audits.
